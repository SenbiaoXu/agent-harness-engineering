# 项目转换技能

将一个项目从某种编程语言完整转换为另一种编程语言。转换后的项目需要在功能和用法上与原项目无缝替换。

## 核心原则

- **功能等价**：转换后的项目需要在生产环境中直接替换原项目，所以每个功能都必须一一对应。使用者在切换后不应遇到功能缺失或行为差异——这意味着相同的输入应该产生语义相同的输出。
- **实现思路对应**：每个模块的实现逻辑尽可能保持一致，这样原项目的维护者在阅读新代码时能快速建立对应关系。当目标语言的惯用做法与源语言差异较大时（如 Python 生成器 vs Go channel），按目标语言的习惯调整，但要确保行为等价。
- **用户确认优先**：语言转换中经常出现"多种实现方式都可行"的情况，不同选择会影响性能、可维护性、生态兼容性。这些取舍应该由了解项目上下文的用户来决定，而不是由你自行判断——即使用户之前给出过通用偏好，遇到新的多选场景时仍应呈现选项。
- **验证不可省略**：功能对比和构建/运行检查是确认转换质量的唯一手段。跳过验证就交付代码，相当于在没有测试的情况下上线——原项目中细微的行为差异（如错误信息格式、边界值处理）很容易在转换中被遗漏。

## 工作流程

每个阶段结束时都有一个**完成门控**——需要满足特定条件才能进入下一阶段。这确保了每个阶段的产出质量，也为回退提供了清晰的检查点。

### 阶段零：确认转换需求

目标：在动手之前，明确转换的范围和关键参数。这一步能避免后续阶段做无用功——比如分析了整个项目才发现用户只想转一个模块，或者做到一半才发现目标语言理解错了。

从用户的请求中提取以下信息，缺少的向用户确认：

1. **源语言与目标语言**：确认要从哪种语言转换到哪种语言。如果用户说的是框架名而非语言名（如"转成 Spring Boot"），推断出对应的语言（Java）并向用户确认。
2. **项目路径**：原项目所在的目录。如果用户没有明确指出，检查当前工作目录是否就是目标项目。
3. **转换范围**：是整个项目还是部分模块。用户有时会说"帮我把认证模块转成 Go"，这时只需要转换指定模块及其依赖。
4. **目标项目位置**：转换后的代码放在哪里。默认建议与原项目同级的新目录（如 `my-project-go/` 与 `my-project/` 并列），也可以由用户指定。
5. **是否保留原项目**：确认原项目不会被删除或覆盖。转换过程中原项目是只读参考，所有新代码写入目标项目目录。

将确认结果简要记录后，进入阶段一。

**完成门控**：源语言、目标语言、项目路径三项必须明确。转换范围和目标位置如果用户未指定，采用默认值（整个项目、同级新目录）并告知用户。

### 阶段一：深度理解原项目

目标：建立对原项目的完整认知地图，为后续转换奠定基础。

1. **项目结构扫描**：列出项目的完整文件结构，识别源代码文件、配置文件、构建脚本、测试文件、文档等。
2. **依赖分析**：读取包管理文件，梳理项目的外部依赖及其用途。
3. **入口点定位**：找到项目的入口文件，理解项目的启动流程。
4. **模块划分**：从入口点出发，逐层追踪引用关系，将项目从业务视角划分为独立的功能模块。对每个模块：
   - 明确模块的职责和对外接口
   - 梳理模块内部的实现逻辑（核心算法、数据流转、状态管理）
   - 标注模块间的依赖关系和调用链
5. **构建与运行方式**：记录原项目的构建命令、运行命令、环境要求、配置方式。
6. **输出阶段一报告**：读取 `references/report-templates.md` 中的「项目分析报告」模板，按格式输出分析结果。

**完成门控**：向用户展示项目分析报告，由用户确认分析是否完整准确。只有用户确认后才能进入阶段二。如果用户指出遗漏或错误，补充分析后重新提交确认。

### 阶段二：逐模块设计转换方案

目标：为每个功能模块制定转换方案，遇到多种选择时由用户决策。

对阶段一识别出的**每个功能模块**，依次完成以下工作：

1. **分析转换要点**：
   - 该模块在源语言中的实现方式
   - 目标语言中是否有直接对应的实现手段
   - 需要做哪些适应性调整

2. **提出转换方案**：
   - 如果只有一种合理的转换方案，简要说明即可
   - 如果有两种或以上可行的转换方案，全部列出，每个方案包含：
     - 方案名称
     - 实现思路（简述）
     - 与原项目的对应关系（功能是否完全等价）
     - 优缺点（性能、可维护性、生态兼容性等）
     - 需要引入的目标语言依赖

3. **等待用户确认**：
   将方案选项呈现给用户。格式示例：

   > 模块 [模块名] 有以下转换方案：
   > - 方案A：[简要描述] — 优点/缺点
   > - 方案B：[简要描述] — 优点/缺点
   >
   > 请选择您倾向的方案。

   **注意**：这里的等待不是形式主义——不同的转换方案会导致完全不同的代码结构、依赖和性能特征，用户需要根据项目实际情况做出取舍。即使用户没有立即回复，也应等待确认后再继续。如果用户在之前已经对某些模块给出了通用偏好（如"全部用最简单的方案"），可以在后续模块中遵循该偏好，但第一次遇到多方案时仍应确认。

4. **记录确认结果**：将用户选择的方案记录下来，作为阶段三的执行依据。

**完成门控**：所有模块的转换方案都经过用户确认后，进入阶段三。如果后续阶段三执行中发现某个模块的设计方案有问题，回退到阶段二重新讨论该模块的方案——不需要重新讨论所有模块。

### 阶段三：执行代码转换

目标：按照用户确认的方案，逐模块将代码转换为目标语言。

1. **项目骨架搭建**：
   - 创建目标语言的项目目录结构
   - 初始化包管理文件和构建配置
   - 设置好基本的目录布局，对应原项目的模块划分

2. **按依赖顺序转换模块**：
   - 优先转换底层、被依赖的模块（工具函数、数据类型定义、配置模块等）
   - 然后转换中间层（业务逻辑、数据处理等）
   - 最后转换顶层（入口文件、API 路由、CLI 命令等）
   - 每转换完一个模块，确保它能在项目内被正确引用

   **并行转换**：如果多个模块之间没有依赖关系（例如两个独立的工具库模块），可以利用 Agent 并行转换以提升效率。并行转换的模块之间不应有 import/引用关系。

3. **转换注意事项**：
   - 保持模块的对外接口与原项目一致（函数名、参数、返回值语义）
   - 如果目标语言的命名惯例不同（如 camelCase vs snake_case），按目标语言惯例调整命名，但在注释中标注对应的原始名称
   - 错误处理方式按目标语言的惯用方式实现，但错误类型和错误信息应与原项目对应
   - 测试文件同步转换
   - 如果转换过程中发现原项目的 bug，在注释中标注但不主动修复——项目转换的职责是等价转换而非改进

4. **配置文件转换**：
   - 将构建脚本、CI 配置、环境配置等转换为目标语言的对应形式

5. **进度记录**（大型项目适用）：
   - 对于模块数量多、跨会话完成的项目，在项目根目录维护一个 `CONVERSION_STATUS.md` 文件，记录每个模块的转换状态（待转换/进行中/已完成）和关键决策摘要
   - 新会话恢复时，读取此文件了解已完成的进度，从下一个未完成的模块继续

**完成门控**：所有模块和配置文件转换完毕后，进入阶段四。

### 阶段四：功能对比验证

目标：确保转换后的项目与原项目功能完全等价。

1. **逐模块功能对比**：
   对阶段一识别的每个功能模块，逐项检查：

   | 检查项 | 说明 |
   |--------|------|
   | 功能覆盖 | 原模块的每个功能是否在新模块中都有对应实现 |
   | 接口一致 | 对外暴露的函数/类/API 是否保持相同的语义 |
   | 行为一致 | 相同输入是否产生相同的输出（允许类型差异，但语义应一致） |
   | 边界情况 | 错误处理、空值处理、异常情况是否都已覆盖 |
   | 配置对应 | 原项目的配置项是否都有对应 |

2. **生成差异报告**：
   如果发现任何不一致，读取 `references/report-templates.md` 中的「功能差异报告」模板，按格式输出。

3. **修复差异**：
   - 对于用户在阶段二确认时明确允许调整/简化的功能，标注为"用户确认的差异"，不需要修复
   - 对于其他差异，逐一修复，确保功能完全对应

4. **向用户汇报验证结果**：
   展示对比结果和已修复的差异，确认用户满意。

**完成门控**：用户确认所有差异已处理完毕后，进入阶段五。如果发现重大功能缺失，回退到阶段三补充转换。

### 阶段五：构建与运行检查

目标：确保转换后的项目可以正常构建和运行。

1. **安装依赖**：在目标项目目录中安装所有依赖。

2. **构建检查**：
   - 执行目标项目的构建命令
   - 解决所有编译错误和警告
   - 确保构建产物正确生成

3. **运行检查**：
   - 启动项目，验证是否能正常运行
   - 如果项目有测试套件，运行全部测试
   - 如果项目有 CLI 或 API，尝试基本操作验证

4. **输出最终报告**：读取 `references/report-templates.md` 中的「转换完成报告」模板，按格式输出。

**完成门控**：构建和运行都通过后，向用户展示最终报告。如果构建或运行失败，在阶段三和阶段五之间迭代修复，直到通过。

## 特殊场景处理

### 大型项目
如果项目模块很多（超过 10 个模块），在阶段二可以将模块分组（如核心模块、辅助模块、测试模块），按组与用户讨论方案，而非逐个模块讨论。但每组内的每个模块仍需逐一确认方案选择。对于跨会话的大型项目，使用阶段三中提到的 `CONVERSION_STATUS.md` 跟踪进度。

### 不支持的特性
如果源语言的某个特性在目标语言中没有直接对应（如 Python 的装饰器在 Go 中无对应），应思考最接近的替代方案，在阶段二中明确说明功能差异，由用户决定是否接受。

### 部分转换
如果用户只需要转换项目的部分模块（而非整个项目），在阶段一仅分析需要转换的部分，但需了解它与其他模块的交互关系，确保转换后接口仍然兼容。

### 错误恢复
转换过程中可能遇到以下问题：
- **依赖不可用**：目标语言的生态中找不到等效依赖。列出差距，让用户选择：用替代库（标注差异）、自行实现、或跳过该功能。
- **编译错误堆积**：先解决底层模块的类型错误，再向上层传播修复。不要试图一次性修复所有错误——自底向上的修复更高效。
- **运行时行为差异**：对比原项目的测试用例输出，定位差异来源。优先修复影响核心功能的差异。
